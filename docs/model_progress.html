Tokenisation Strategy

Each incident is encoded as a fixed 6-token tuple. The model learns the language of urban emergencies by predicting the next token in a per-station sequence ordered chronologically over 15 years.

Incident tuple (6 tokens per event)

DT_x → STATION_x → WARD_x → TYPE_x → STOP_x → PROP_x

DT = time-gap to previous incident  |  STATION = responding station  |  WARD = incident ward  |  TYPE = incident category  |  STOP = stop code  |  PROP = property type

Window context prefix (8 tokens, prepended to each training window)

<BOS> REGIME TEMP_x RAIN_x WIND_x DOW_x HOUR_x MONTH_x

GAP bucket boundaries

Derived from the actual inter-incident gap distribution. Bins are logarithmically spaced to capture both rapid successive calls and quiet overnight periods.

Output Schema — forecast_24h.json (for map team)

Phase 6 inference will aggregate per-station rollouts into this flat ward-level JSON. This is the contract between the model pipeline and the map frontend. 

{
  "generated_at": "2024-01-12T18:00:00Z",
  "forecast_date": "2024-01-13",
  "weather_context": {
    "temp_c": 4, "rain_mm": 0, "wind_kmh": 12
  },
  "wards": [
    {
      "ward_name":   "Gospel Oak",           // matches GeoJSON ward boundary name
      "borough":     "Camden",
      "lat":         51.554,               // WGS84
      "lon":        -0.163,
      "risk_score":  0.72,                 // 0.0–1.0  →  heatmap opacity
      "hourly_risk": [0.1, 0.1, 0.2, ...], // 24 values  →  hour slider
      "top_type":    "false_alarm",         // dominant type  →  map colour
      "type_breakdown": {
        "dwelling_fire":   0.08,
        "outdoor_fire":    0.05,
        "false_alarm":     0.52,
        "special_service": 0.35
      }
    }
  ]
}

GeoJSON ward boundaries: London Datastore — Statistical GIS Boundary Files. Coordinate reference: WGS84. Colour coding: Dwelling Fire = red · Outdoor Fire = orange · False Alarm = grey · Special Service = blue.
